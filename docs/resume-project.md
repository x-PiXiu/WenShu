## 版本一：精简版（适合一页简历，4-6 行）

### 文枢 · 藏书阁 — 全栈 RAG 知识库系统

独立设计并开发基于 RAG 的智能知识库系统，支持 PDF/DOCX/MD 等多格式文档解析与向量化索引。核心实现混合检索引擎（向量语义检索 + BM25 关键词检索 → RRF 融合排序），检索准确率较单一向量检索提升显著。集成 LangChain4j 构建 13 个 Agent 工具（知识检索、Web 搜索、翻译、文档对比等），支持 LLM 流式输出与多模型热切换。前端采用 Vue 3 + TypeScript，后端基于 Javalin 6.x + SQLite，实现 Prompt 统一管理、闪卡学习、博客发布、LLM 调用监控等完整功能模块。项目已开源并发布可下载即用的 Release 包。

## 版本二：标准版（适合详细简历，带技术栈标签）

### 文枢 · 藏书阁 — 全栈 RAG 智能知识库

**项目描述**：独立设计开发的 RAG（检索增强生成）知识库系统。用户上传文档后，系统自动完成解析、分块、向量化索引，通过自然语言进行精准问答，每个回答附来源引用和置信度评分。

**技术栈**：Java 17 / Javalin 6.x / LangChain4j 1.14 / Vue 3 / TypeScript / SQLite / Apache Tika / PDFBox

**核心工作**：

- **混合检索引擎**：设计并实现向量语义检索 + BM25 关键词检索的双重召回机制，通过 RRF（Reciprocal Rank Fusion）融合排序，兼顾语义理解和精确匹配，构建自动化评估体系（Recall@K / MRR）
- **Agent 工具调用**：基于 LangChain4j @Tool 注解体系开发 13 个可调用工具（知识库检索、Web 搜索、翻译、计算、文档对比等），LLM 可根据用户意图自主决策工具调用链
- **Prompt 统一管理**：设计 PromptRegistry 单一事实来源架构，将核心功能、工具、智能体提示词统一注册管理，支持后台热更新与持久化；AgentStore 通过 prompt_key 引用而非内嵌文本，实现提示词与智能体的解耦
- **流式问答系统**：基于 SSE 实现逐 Token 流式响应，支持多轮对话上下文、长期记忆（重要性评分 + 衰减机制）、多智能体切换
- **多模型热切换**：运行时切换 LLM / Embedding 模型和向量存储后端（InMemory / Chroma / Milvus），零停机生效
- **闪卡学习系统**：AI 生成深度闪卡（直接回答 + 深度解释双层结构），内置 SM-2 间隔重复算法，支持 PDF 导出（PDFBox）
- **LLM 可观测性**：构建调用追踪系统，实时监控 Token 消耗、延迟分布、模型使用统计，配合 RAG 质量评估仪表盘
- **全栈工程化**：前端 Vue 3 SPA（Naive UI），后端 RESTful API 70+ 端点，SQLite 共享连接池，Javalin 静态文件服务，Maven Assembly 一键打包发布

**项目成果**：开源发布，支持下载即用（Fat JAR + 启动脚本），兼容 A2A / MCP 协议

## 版本三：面试口述话术（2 分钟版本）

> 这个项目是我在研究 RAG 技术落地时，发现大多数开源方案要么是简单的 Demo 级别，要么部署门槛很高。所以我自己从零设计了一个完整的、可部署的个人 AI 知识库。
>
> 核心亮点是**混合检索引擎**。单纯向量检索擅长"意思相近"但容易漏掉精确关键词，单纯 BM25 擅长"字面匹配"但理解不了语义。我把两者做双重召回，再用 RRF 算法融合排序，效果比单一检索好很多。我还做了一个自动化评估体系来量化这个提升。
>
> 第二个亮点是 **Agent 工具调用**。基于 LangChain4j 的 @Tool 体系开发了 13 个工具，LLM 可以根据用户意图自主决定调不调用、调哪些。比如用户问"帮我对比这两篇文档的差异"，LLM 会自动调用文档对比工具。
>
> 第三个是 **Prompt 统一管理**。我设计了一个 PromptRegistry 作为唯一事实来源，所有提示词——核心功能的、工具调用的、智能体人设的——都在这一个注册表里管理，支持后台编辑热更新。Agent 只存一个 prompt_key 引用，不需要每个 Agent 各自维护一份完整提示词文本。
>
> 整个系统是 Java + Vue 3 的全栈架构，后端 70 多个 API 端点，前端是完整的 SPA。已经开源发布，用户下载解压就能用，不需要装 Maven 或者 Node.js。

## 面试高频问题准备

### Q: 为什么不用 Spring Boot 而用 Javalin？

Javalin 是轻量级 HTTP 框架，启动快、内存占用低、API 简洁。这个项目的核心是 RAG 检索和 LLM 集成，不需要 Spring 的 IoC 容器、AOP、自动装配这些重量级特性。Javalin 让整个后端只有一个主类，路由定义直观，部署产物是一个 100MB 的 Fat JAR 而不是 Spring Boot 那种还要内嵌 Tomcat 的方案。

### Q: 混合检索中 RRF 融合的具体实现？

两路检索并行执行：向量检索通过 Embedding 相似度取 Top-K，BM25 通过关键词匹配取 Top-K。然后对两路结果分别按排名生成 RRF 分数：`score = 1 / (k + rank)`，其中 k=60 是阻尼常数。最终按合并后的 RRF 分数排序。这样排名靠前的结果无论来自哪路，都能获得较高分数。

### Q: Prompt 统一管理解决了什么问题？

之前每个 Agent 在 SQLite 中存一份完整的 system_prompt 文本，和 PromptRegistry 中的提示词完全脱离。问题是：默认 Agent 的提示词同时存在于两处，可能不一致；自定义 Agent 的提示词在 PromptRegistry 管控之外，无法热更新。改造后 Agent 只存 prompt_key 引用，实际文本统一由 PromptRegistry 管理，支持后台编辑、即时生效、持久化到 config.json。

### Q: Agent 工具调用的技术实现？

基于 LangChain4j 的 @Tool 注解，在 Java 类中标注工具方法。ToolEngine 在启动时通过反射提取所有工具的名称、描述、参数 schema，注册到 LLM 的 tool 列表中。LLM 返回工具调用请求时，ToolEngine 根据 function name 路由到对应方法执行，结果回传给 LLM 继续生成。支持多轮工具调用链。

### Q: SQLite 作为生产存储有什么考量？

使用 WAL 模式提升并发读写性能，配合 DatabasePool 共享连接池（6 连接）管理。知识库场景是个人/小团队使用，SQLite 的单文件、零部署、内嵌特性非常匹配。如果未来需要水平扩展，可以切换到 PostgreSQL，因为只用了标准 SQL。
